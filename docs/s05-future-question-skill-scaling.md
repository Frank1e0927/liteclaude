# s05 Future Question: Skill 数量增长后的 Prompt 膨胀

> 这是 `s05 skill loading` 在教学实现完成后留下的一个后续问题，不影响当前最小实现，但在真实工程里必须考虑。

---

## 问题描述

当前项目在 [agent.ts](C:/Users/ROG/Desktop/files/agent.ts) 中通过 `getSkillSummaries()` 把所有本地 skill 的：

- `name`
- `description`

拼接进 `systemPrompt`。

这种做法适合教学，因为它简单、直接、容易验证 `s05` 的核心链路：

- 模型先看到技能目录
- 再决定是否调用 `load_skill`
- 被调用时才加载技能正文

但它也带来一个后续问题：

**如果本地安装的 skill 越来越多，system prompt 中的技能目录也会越来越长。**

---

## 为什么这是个问题

当 skill 数量持续增加时，会出现这些风险：

1. 输入 token 增加
2. 每轮请求重复发送技能目录，成本上升
3. 请求延迟增加
4. 无关 skill 摘要增多，模型更容易被干扰
5. 不同 skill 的描述如果相似，模型的选择会变得不稳定

也就是说，`s05` 虽然已经解决了“不要把所有 skill 正文塞进 prompt”这个大问题，但还没有彻底解决“skill 摘要层本身也会膨胀”的问题。

---

## 当前实现为什么仍然合理

当前实现仍然是合理的教学版本，因为：

- 它优先解决的是“正文延迟加载”问题
- skill 摘要比 skill 正文短很多
- 在 skill 数量较少时，这种实现成本很低、结构清晰
- 它能帮助理解 `s05` 的核心设计，而不会过早引入检索复杂度

所以，这不是“当前实现错了”，而是：

**当前实现适合教学和小规模 skill 集合，但未来需要扩展。**

---

## 未来可以考虑的改进方向

### 方案 1：限制摘要数量

只把前 `N` 个最重要或最常用的 skill 摘要放进 `systemPrompt`。

优点：

- 实现简单
- 能立刻限制 prompt 膨胀

缺点：

- 不够智能
- 容易漏掉真正相关但不在前 `N` 个里的 skill

### 方案 2：按任务检索相关 skill

先根据当前任务做本地匹配，只把最相关的少量 skill 摘要注入 `systemPrompt`。

优点：

- 更节省 token
- 更符合真实工程使用方式

缺点：

- 需要额外设计匹配逻辑
- 复杂度高于教学版

### 方案 3：分层技能目录

`systemPrompt` 里先只放技能类别，不放全部具体技能。
当模型需要时，先搜索类别或子技能，再加载正文。

优点：

- 更适合大量 skill
- 结构更清晰

缺点：

- 需要多一层工具或目录设计

### 方案 4：引入 skill 搜索工具

除了 `load_skill` 之外，再加入 `search_skills` 工具。

流程变成：

1. 模型先调用 `search_skills`
2. 获得候选 skill
3. 再调用 `load_skill`

优点：

- 更像真实产品设计
- 不需要把完整技能目录长期留在 prompt 中

缺点：

- 超出当前 `s05` 教学最小实现范围

---

## 当前阶段的结论

在当前教学阶段，我们接受以下折中：

- 把所有 skill 的简短摘要放进 `systemPrompt`
- 不把 skill 正文放进去
- 用 `load_skill` 按需加载正文

这个折中足以帮助理解 `s05` 的核心机制。

但在 skill 数量继续增长时，应该把“全量摘要注入”升级为“筛选后注入”或“先检索再加载”。

---

## 后续建议

如果后面继续演进这个 TS 项目，建议把这个问题放在 `s05` 的后续优化列表中，优先级高于“增加更多示例 skill”。

原因是：

- skill 数量越多，这个问题越明显
- 这是 prompt 成本与可扩展性的结构性问题
- 它比新增 skill 内容更影响系统整体质量
